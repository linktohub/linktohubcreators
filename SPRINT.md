## Task 1 — Stripe Connect Onboarding CRITICAL

**Why:** The "Connect Stripe" button in `/dashboard/payouts` has `onClick={() => {}}` — it does nothing. `paymentIntents.create` in `/api/checkout/intent` doesn't use `transfer_data` or `application_fee_amount`, so even when a customer pays, 100% of the money sits in the platform Stripe account and the creator never sees a dollar. This is the single biggest blocker.

**What to build:**
- `src/app/api/stripe/connect/route.ts` — POST handler: creates a Stripe Express account for the creator (if none exists), saves `stripe_account_id` to `creators` table, generates an `AccountLink` for onboarding, returns the URL
- `src/app/api/stripe/connect/return/route.ts` — GET handler Stripe redirects to after onboarding; calls `stripe.accounts.retrieve()`, sets `stripe_account_enabled = true` in `creators` if `charges_enabled`
- `src/app/dashboard/payouts/page.tsx` — replace `onClick={() => {}}` with a `<form action="/api/stripe/connect">` POST that redirects the browser to the Stripe onboarding URL; add a "Refresh status" button that re-checks `stripe_account_enabled` after return
- `src/app/api/checkout/intent/route.ts` — fetch `stripe_account_id` for the `creatorId`; if present, add `transfer_data: { destination: stripe_account_id }` and `application_fee_amount: Math.round(totalCents * 0.10)` to the PaymentIntent

**Done when:**
- A creator clicks "Connect Stripe", completes Stripe Express onboarding, and the dashboard shows "Payouts enabled"
- A test purchase routes 90% to the creator's connected account automatically (verifiable in Stripe dashboard)

## Task 2 — Shipping Address Collection + Gelato Fulfillment CRITICAL

**Why:** The webhook at `src/app/api/checkout/webhook/route.ts` creates an `orders` DB record on `payment_intent.succeeded` but never calls Gelato. Merch buyers get charged and receive nothing. Additionally, the cart checkout in `storefront-client.tsx` collects an email but no shipping address — so even if the webhook tried to call Gelato, it would fail.

**What to build:**
- `src/app/[username]/storefront-client.tsx` — add a shipping address form step in the cart drawer (shown only when `cart` contains items with `type === "merch"`). Fields: firstName, lastName, addressLine1, city, postCode, country. Pass `shippingAddress` and `shippingName` in the `/api/checkout/intent` POST body; store them in PaymentIntent `metadata` (as JSON string).
- `src/app/api/checkout/intent/route.ts` — accept `shippingAddress` and `shippingName` in the request body and store in `metadata.shipping_address` and `metadata.shipping_name`
- `src/app/api/checkout/webhook/route.ts` — after inserting the order, check if any item in `metadata.items` maps to a `type === "merch"` product (fetch from Supabase). If so, call `createGelatoOrder` from `src/lib/gelato.ts` using the stored shipping address, creator's Gelato product/variant UIDs, and the internal order ID. Update the order row with `pod_order_id`, `tracking_number`, `tracking_url`, `status: "fulfilled"`.

**Done when:**
- A test merch purchase triggers a real Gelato order (visible in Gelato dashboard) with the buyer's address
- The `orders` table row is updated with a Gelato order ID and fulfillment status

## Task 3 — Digital Product Delivery via Secure Download Link CRITICAL

**Why:** When a digital product is purchased, the webhook records the order but never delivers the file. Buyers pay and get a "Order placed! 🎉" toast with no follow-up. `src/app/dashboard/digital/` shows products can have a `file_url` column — but there is no download route, no access control, and no post-purchase redirect or email.

**What to build:**
- `src/app/api/download/[token]/route.ts` — GET handler: validate a signed JWT or UUID token stored in `purchase_tokens` table (columns: `id`, `order_id`, `product_id`, `buyer_email`, `expires_at`). If valid, generate a Supabase Storage signed URL for `products.file_url` and redirect to it (or proxy the download). Tokens expire after 72 hours or 5 downloads.
- `src/app/api/checkout/webhook/route.ts` — for each digital item in the order: insert a row into `purchase_tokens` with a UUID token and 72h expiry. Return the download URL(s) in a Supabase edge function call or store on the order row (`download_urls` JSONB column).
- `src/app/[username]/storefront-client.tsx` — after `confirmCardPayment` succeeds for a digital-only cart, poll `/api/orders/[paymentIntentId]` for 10s and redirect to a `/download?token=...` page, or show download links inline in the success state.
- Supabase migration: `purchase_tokens` table with RLS disabled (token IS the auth); `orders.download_urls` JSONB column.

**Done when:**
- A test digital product purchase produces a working download link the buyer can click within 60 seconds of payment
- Link expires after 72h (verifiable by checking `expires_at`)

## Task 4 — Product Edit Pages HIGH

**Why:** Creators can only delete products, not fix a typo in the title or adjust the price. This creates churn during early onboarding when creators are iterating on their store. The products list at `src/app/dashboard/products/products-client.tsx` has no edit link; the `src/app/dashboard/products/` directory has `create/` and `delete-button.tsx` but no `[id]/` edit route.

**What to build:**

- `src/app/dashboard/products/[id]/edit-client.tsx` — client component with the same fields as the create form (name, description, price, active toggle). On submit, `supabase.from("products").update(...)`. Redirect to `/dashboard/products` on success.
- `src/app/dashboard/products/products-client.tsx` — add an edit icon/link next to each product row pointing to `/dashboard/products/[id]`
- Apply the same pattern to `subscription_tiers` (edit tier) and `events` (edit event) — same page structure, one hour of work.

**Done when:**
- A creator can change a product's name, price, or description and see the update reflected on the storefront immediately

## Task 5 — Gate AI Chat Behind Fan Subscription HIGH

**Why:** `src/app/api/ai/chat/route.ts` has no auth check at all — any anonymous visitor can send unlimited messages to Claude on the platform's API bill. This is giving away the most defensible premium feature for free. AI Chat is meant to be a subscription perk.

**What to build:**
- `src/app/api/ai/chat/route.ts` — add a Supabase auth check at the top. If no session, return `{ error: "Subscribe to chat", gate: true }`. If session exists, check `fan_subscriptions` for an active row matching `creator_id` with `status = "active"`. If no active subscription, return `{ error: "Subscribe to chat", gate: true }`.
- `src/app/[username]/storefront-client.tsx` — in the `sendChat` handler, handle `gate: true` response by: clearing the chat input, showing a toast "Subscribe to unlock AI chat", and switching `activeTab` to `"subscribe"` so the fan sees the membership tiers immediately.
- Add a `data-testid="chat-gate"` message in the chat UI: "Chat with [creator] is a members-only perk — subscribe below."

**Done when:**
- An unauthenticated visitor cannot send a chat message (gets redirected to subscribe tab)
- A fan with an active subscription can chat normally
- No free Claude API calls are triggered by anonymous visitors
